# issue-triage

Triage is classification backed by evidence, not label bookkeeping.
Every triaged item carries exactly one category — **bug** or
**enhancement** — and exactly one state:

- `needs-triage` — awaiting evaluation
- `needs-info` — blocked on the reporter
- `ready-for-agent` — fully specified; an unattended agent can execute
- `ready-for-human` — needs judgment, access, or design a human owns
- `wontfix` — will not be actioned, with the reason recorded

Untriaged items enter at `needs-triage`; `needs-info` returns there
when the reporter replies. Conflicting labels get flagged to the
maintainer, never silently resolved. Every comment posted to the
tracker during triage opens with:

```
> *This was generated by AI during triage.*
```

## Surfacing what needs attention

Present three buckets, oldest first, with counts and a one-line
summary per item: never-triaged items, items sitting in
`needs-triage`, and `needs-info` items with reporter activity since
the last triage note. Let the maintainer pick.

## Triaging an item

1. **Gather.** Read the full thread — body, comments, labels, dates —
   and any prior triage notes, so resolved questions are never
   re-asked. Then two codebase checks: search for an existing
   implementation of the requested behavior (by domain concept, not
   the reporter's wording, reporting where the search looked), and
   check recorded wontfix rationale for a prior rejection of the same
   idea.
2. **Verify the claim.** For a bug, reproduce it from the reporter's
   steps; for an enhancement, confirm the gap is real. Report the
   result: confirmed with the code path, failed to reproduce, or too
   vague to attempt — which is the strongest `needs-info` signal
   there is.
3. **Recommend.** Present category, state, and the evidence, then
   wait for the maintainer's direction — posting to a tracker is
   outward-facing, and nothing is posted before it.
4. **Apply the outcome.**
   - `ready-for-agent` — attach a brief built with the
     define-work-items discipline, casting the implementer as an
     unattended agent: zero open judgment calls, binary acceptance
     criteria.
   - `ready-for-human` — the same brief, plus an explicit note on why
     it cannot be delegated (judgment calls, external access, design
     decisions, manual testing).
   - `needs-info` — triage notes recording what is established so far
     and the specific questions outstanding; "please provide more
     info" is not a question.
   - `wontfix` — close with the reason. Already-implemented requests
     point at where the behavior lives; rejected enhancements get the
     rationale written down where future triage will find it, so the
     same request is never re-litigated from scratch.

## Direct orders

"Move #42 to ready-for-agent" is an instruction, not a triage
request: confirm the exact actions about to be taken — labels,
comment, close — then apply them without re-running the pipeline,
offering to attach a brief if none exists.
